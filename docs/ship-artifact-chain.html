Section 01 Section 01

Ý nghĩa của từng pack What each pack is actually for

Mỗi artifact nên trả lời đúng một câu hỏi. Nếu một artifact làm quá nhiều việc, flow sẽ bắt đầu rối và AI sẽ dễ quay lại khám phá repo từ đầu. Each artifact should answer one question well. If one artifact tries to do too much, the flow becomes messy and AI starts rediscovering the repo from scratch.

01

master-package

Định nghĩa feature cần build gì. Defines what the feature needs to deliver.

  • chứa scope đã approve và quyết định sản phẩm holds approved scope and product decisions
  • thường bao gồm master-spec.md, knowledge.md, api-contract.md khi có usually includes master-spec.md, knowledge.md, and api-contract.md when present
  • không nên viết theo kiểu implementation plan should not read like an implementation plan
02

kh-search-brief

Thu hẹp search space trước khi explore repo. Reduces the search space before repo exploration starts.

  • chỉ là hint, không phải source of truth it is a hint layer, not a source of truth
  • nên chứa candidate modules, keywords, touch points, integrations it should contain candidate modules, keywords, touch points, and integrations
  • nằm ở .planning/<feature>/kh-search-brief.md it lives at .planning/<feature>/kh-search-brief.md
03

repo-exploration

Verify code hiện tại thực sự đang thế nào. Verifies what the current code actually looks like.

  • confirm touch points, reusable patterns, constraints, conflicts confirms touch points, reusable patterns, constraints, and conflicts
  • đây là nơi code reality bắt đầu thắng KH hint this is where code reality starts to override KH hints
  • nằm ở .planning/<feature>/repo-exploration.md it lives at .planning/<feature>/repo-exploration.md

  • nên đủ mạnh để AI codegen từ delta trước should be strong enough for AI to codegen from the delta first
  • chỉ mở lại KH hoặc repo-exploration khi cần chi tiết hẹp KH and repo-exploration should only be reopened for narrow detail checks
Section 02 Section 02

Build delta cho từng platform How to build a delta for each platform

Delta được build sau khi đã có package, KH brief, và repo exploration. AI nên đọc chúng theo đúng thứ tự đó, rồi mới viết delta theo platform. Delta is built after package, KH brief, and repo exploration already exist. AI should read them in that order before writing a platform delta.

Web

clad-ui first
  • route/page/layout nào bị ảnh hưởng which routes, pages, or layouts are affected
  • component nào reuse, extend, create which components to reuse, extend, or create
  • state, data, tracking, socket, localStorage behavior state, data, tracking, socket, and localStorage behavior

Android / iOS

UI + ownership
  • screen, navigation, owner modules screens, navigation, and owner modules
  • ViewModel/store/interactor/repository touches ViewModel, store, interactor, or repository touches
  • state transitions và native UI patterns cần follow state transitions and native UI patterns to follow

Backend

behavior first
  • handler/service/job nào bị ảnh hưởng which handlers, services, or jobs are affected
  • business rules, validation, auth, side effects business rules, validation, auth, and side effects
  • observability, rollback, event impact observability, rollback, and event impact

QE / QC

quality flow
  • coverage, edge cases, regression focus coverage, edge cases, and regression focus
  • automation foundation hoặc manual quality strategy automation foundation or manual quality strategy
  • gaps nào vẫn phải giữ visible which gaps must stay visible

Cách hướng dẫn AI build delta How to steer AI when building a delta

Đừng nói chung chung kiểu “generate delta”. Hãy ép AI theo đúng chuỗi artifact và output shape của platform. Do not ask vaguely for “a delta”. Force AI to use the artifact chain and the target platform shape.

Đọc theo thứ tự này: 1. master-package 2. kh-search-brief 3. repo-exploration Sau đó viết -delta.md theo đúng platform contract. Yêu cầu: - bám vào verified repo reality - nêu rõ change surface - nêu rõ reuse / extend / create - giữ fallback refs chỉ cho chi tiết hẹp - nếu delta chưa đủ mạnh để codegen thì giữ trạng thái DraftRead in this order: 1. master-package 2. kh-search-brief 3. repo-exploration Then write -delta.md using the platform contract. Requirements: - stay grounded in verified repo reality - make the change surface explicit - make reuse / extend / create explicit - keep fallback refs only for narrow details - if the delta is not strong enough for codegen, keep it in Draft
Rule quan trọng: delta không phải PRD thứ hai. Delta phải là implementation contract cho platform đó. Important rule: delta is not a second PRD. Delta should become the implementation contract for that platform.
Section 03 Section 03

Chạy implement như thế nào How implementation runs afterward

Sau khi có đủ kh-search-brief, repo-exploration, và delta, phase tiếp theo là /implement. Nó có đúng hai mode. After kh-search-brief, repo-exploration, and delta exist, the next phase is /implement. It has exactly two modes.

mode = full

delta-first

Đi thẳng từ delta tới final codegen. Go straight from delta to final codegen.

1

Đọc full delta Read the full delta

Coi delta là implementation contract chính. Treat the delta as the primary implementation contract.

2

Codegen một pass Codegen in one pass

Chỉ mở repo-exploration hoặc kh-search-brief khi cần chi tiết hẹp. Only open repo-exploration or kh-search-brief for narrow detail checks.

3

Verify và report Verify and report

Chạy lint/tests/typecheck khi feasible rồi trả changed files và unresolved risks. Run lint, tests, or typecheck when feasible, then return changed files and unresolved risks.

mode = multi

phase-gated

Tạo plan file, chạy từng phase, và dừng cho user review. Create a plan file, run one phase at a time, and pause for user review.

1

Generate implementation-plan.md Generate implementation-plan.md

Plan chỉ chia execution theo phase, không redefine scope của delta. The plan only splits execution into phases; it does not redefine delta scope.

2

Chạy đúng một phase Run exactly one phase

Implement phase hiện tại, verify phase hiện tại, rồi cập nhật plan state. Implement the current phase, verify the current phase, then update plan state.

3

Dừng và chờ resume Pause and wait for resume

Sau mỗi non-final phase phải in rõ: “If you want me to continue to the next phase, reply: continue”. After every non-final phase, print: “If you want me to continue to the next phase, reply: continue”.

Rule ngắn gọn: full dùng khi delta đã đủ mạnh để codegen một lèo. multi dùng khi muốn chia phase, checkpoint, và giữ user trong vòng review. Short rule: use full when the delta is already strong enough for straight-through codegen. Use multi when you want phases, checkpoints, and explicit user review between them.

Nếu phải nhớ một câu duy nhất: master-package nói feature cần gì, KH brief nói nên nhìn đâu trước, repo exploration nói code hiện tại ra sao, delta nói platform nên implement thế nào, còn implement mode quyết định cách chạy codegen. If you only remember one sentence: master-package says what the feature needs, KH brief says where to look first, repo exploration says what the code looks like, delta says how the platform should implement it, and implement mode decides how codegen should run.
